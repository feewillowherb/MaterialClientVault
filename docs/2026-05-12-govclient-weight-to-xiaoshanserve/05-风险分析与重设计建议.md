# 05 - 风险分析与重设计建议

## 当前架构风险评估

### 高风险

#### R1: 稳定性检测算法不可靠

**现象**: 仅通过"连续 5 次 Timer tick 重量不再超过当前最大值"判断稳定，无法检测真实波动。

**具体问题**:
- **单向递减**: 重量从 5000→4990→4980→4970→4960→4950，5 次后判定稳定，但实际仍在下降
- **无范围判断**: 重量在 5000±100 波动但始终低于 5000 时，不会触发递减
- **Timer 抖动**: `System.Timers.Timer` 在 UI 线程阻塞时实际间隔不稳定

**证据**: `Form1.cs:527-547`

#### R2: 二次转发增加故障点

**现象**: GovClient → XiaoShanServe → 政府平台，经过两个独立进程。

**影响**: XiaoShanServe 进程崩溃或数据库损坏时，即使 GovClient 正常也无法上报。

**证据**: `ServicePushWeightToGovApi.cs` 上报到 XiaoShanServe，`ExplortStatisticBgService` 再转发到政府平台。

#### R3: 授权检查与业务逻辑耦合

**现象**: 每次上报前检查 RSA 授权时间，过期则终止所有上报。

**影响**: 授权到期不仅影响新数据，还会阻塞已入库未上报的历史数据。

**证据**: `ServicePushWeightToGovApi.cs:221-230`

### 中风险

#### R4: 图片路径硬编码

**现象**: `GetImages()` 使用 `AppDomain.CurrentDirectory + "\\" + item` 拼接路径。

**影响**: 工作目录变化时图片读取失败，导致 `FailReason = 数据缺失` 并标记为"不再上传"（不可恢复）。

**证据**: `ServicePushWeightToGovApi.cs:413`

#### R5: 同步状态管理脆弱

**现象**: `FailCount >= 9 && FailReason != 网络` 就标记为"不再上传"，但 `FailReason` 可能被覆盖为"未知"。

**影响**: 非网络类异常累计 9 次后数据永久丢失。

**证据**: `ServicePushWeightToGovApi.cs:455-456`

#### R6: 无进出场匹配

**现象**: 每次 `InOutType` 固定为 `0`（进场），无出场地称重。

**影响**: 无法计算净重（毛重 - 皮重），政府平台无法获得准确物料重量。

**证据**: `Form1.cs:599` — `InOutType = (int)eInOutType.进场地称重`

### 低风险

#### R7: WinForms 架构限制

**现象**: 使用 `Application.DoEvents()`、`BeginInvoke`、全局静态事件。

**影响**: 难以单元测试、难以扩展、UI 线程阻塞风险。

#### R8: 废弃代码残留

**现象**: `ServicePushWeightToGovApi.cs` 中有大量注释掉的 HTTP 状态码判断代码（约 60 行）。

#### R9: 数据库访问层代码生成

**现象**: `bRePushWeight`、`BaseDAL` 等由 CodeSmith 生成，使用原始 SQL 拼接。

## 与 MaterialClient 的核心差异

| 维度 | GovClient (旧) | MaterialClient (新) | 重新设计影响 |
|------|---------------|-------------------|-------------|
| **架构** | WinForms + Timer + 事件 | Avalonia + Rx.NET + DDD | 全面重写 |
| **稳定性检测** | 计数器 (5×2s) | 时间窗口分析 (3000ms) | 可直接迁移 |
| **数据模型** | mRePushWeight (单表) | WeighingRecord + Waybill (关系型) | 需要重新设计 |
| **进出匹配** | 无 | 自动 + 手动匹配 | 核心新增功能 |
| **同步方式** | SQLite + Timer 轮询 | Refit HTTP + IsPendingSync | 简化为直连 |
| **中转层** | 需要 XiaoShanServe | 直连远程平台 | 可选去中转 |
| **状态管理** | 字段变量 | BehaviorSubject + 状态机 | Rx 模式 |
| **并发安全** | Monitor 锁 | Rx 调度器 | 无需手动锁 |
| **车牌识别** | CaptureDevice 回调 | ILocalEventBus 事件 | 事件总线模式 |
| **配置管理** | App.config + ConfigurationManager | SettingsService + ILocalEventBus | 热更新 |

## 重新设计方向

### 方向 1: 迁移到 MaterialClient 架构

**目标**: 利用 MaterialClient 已有的成熟架构替代 GovClient。

**具体方案**:

1. **保留 XiaoShanServe 的接入码验证机制**，在 MaterialClient 的 `IMaterialPlatformApi` 中增加萧山模式适配
2. **迁移 DiBangScaleReceiver** 的多协议解析能力到 `TruckScaleWeightService`
3. **复用 MaterialClient 的稳定性检测**（`CreateStabilityStream`），替换计数器方案
4. **如果需要保留 XiaoShanServe 中转层**，在 `IMaterialPlatformApi` 中增加转发端点

### 方向 2: 最小化改进（保留现有架构）

**目标**: 不改变整体架构，仅修复关键问题。

**具体方案**:

1. **替换稳定性算法**:
   - 引入滑动窗口（最近 N 个 Timer tick 的重量值）
   - 判断标准: `max - min <= threshold && 数据点 >= M`
   - 稳定值: `(max + min) / 2` 替代最大值

2. **增加出场地称重**:
   - 增加"收料/发料"切换 UI
   - `InOutType` 根据选择动态设置

3. **增加进出场匹配**:
   - 在本地 SQLite 中增加配对逻辑
   - 或直接将匹配逻辑放到 XiaoShanServe 端

4. **修复同步可靠性**:
   - 图片路径使用绝对路径
   - "不再上传"状态需要人工恢复机制
   - 增加同步仪表盘 UI

### 方向 3: 去中转直连政府平台

**目标**: 去掉 XiaoShanServe 中转层，GovClient 直连政府 API。

**具体方案**:

1. 在 `ServicePushWeightToGovApi.Start()` 中增加"萧山直连"模式
2. 直接 POST 到 `http://191.12.15.58:8899/sapi/v1/inoutRecord/save`
3. XiaoShanServe 仅作为数据查看和管理后台
4. 减少故障点，简化部署

## 推荐方案

**建议采用方向 1（迁移到 MaterialClient 架构）**，理由：

1. MaterialClient 已有完整的稳定性检测、状态机、匹配、同步机制
2. GovClient 的核心功能（串口读取、车牌识别、重量上报）在 MaterialClient 中都有对应实现
3. 长期维护两套代码成本高，统一到 MaterialClient 更合理
4. MaterialClient 的 Rx 架构更易测试、更易扩展

如果无法立即全面迁移，可先执行**方向 2**（最小化改进）作为过渡方案。
