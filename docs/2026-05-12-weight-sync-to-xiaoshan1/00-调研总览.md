## 重要前提

- GovClient 是**上传称重记录**，而不是帮当 waybill，所以**不需要匹配**。
- 仅包含同步到 `networkCompany.萧山1` 的功能，不涉及电信、联通模式。
- 可能需要修改 LRP（车牌识别）的代码，需要下载 LRP 的图片。
- MaterialClient 已有完整的 `HikvisionLprService`，可直接复用。

## 扩展需求（低优先级）

服务端需要知道**客户端在线情况**和**设备在线情况**（地磅、相机等）。这是一个独立于核心同步流程的监控能力，不影响称重数据上报的主链路。

## 关键发现

1. **MaterialClient 已有海康威视车牌识别实现**（`HikvisionLprService`），可以直接复用，但当前回调中**未保存图片**，需要补充。
2. **MaterialClient 已有完整的附件管理体系**：`AttachmentFile` 实体 + `AttachType` 枚举 + `AttachmentService` + `PathManager` 路径管理 + OSS 同步。LRP 图片可**直接复用**该体系，仅需新增 `LprCapturePhoto = 4` 枚举值，无需自建图片存储机制。
3. **称重功能相对独立**，主要涉及串口通信（`WeightDevice.cs` / `DiBangScaleReceiver.cs`），与 MaterialClient 的 `TruckScaleWeightService` 对应。
4. **萧山1 同步的特殊性**：不走心跳、MsgAction 为 `"post"`、API 地址为 `govXiaoShan` 配置项、数据模型包含 `carType`/`deviceID`/`siteType`/`goodsWeight` 字段。
5. **图片同步链路已打通**：LRP 图片通过 `AttachmentFile` 持久化（相对路径），同步服务通过 `PathManager.ToAbsolutePath()` 还原后 Base64 编码上传，与 GovClient 的 `SnapImages` 逗号分隔路径模式等价但更规范。
6. **已有链路分析**：`docs/2026-05-12-govclient-weight-to-xiaoshanserve/` 已完成完整的数据链路分析，本文件夹聚焦于移植方案设计。
7. **在线状态上报已有部分基础**：GovClient 通过 `ServiceTelnet` 检测网络连通性，`Form1` 用 3 秒定时器跟踪设备在线状态；MaterialClient 已有 `LprDeviceOnlineStatusService` 统一查询设备在线状态，`MinimalWebHostService` 提供回调 API 端点。但两端均缺少**客户端→服务端的主动在线状态上报**机制。
