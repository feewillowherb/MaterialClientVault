# 03 - 最小改动移植方案

## 1. 方案总览

采用**增量叠加**策略：在 MaterialClient 现有架构上新增称重同步模块，复用已有的海康威视服务、串口通信和配置管理，最小化改动量。

```
改动范围（最小集合）：

MaterialClient.Common/
├── Services/
│   ├── Hikvision/HikvisionLprService.cs   ← 修改：回调中保存图片
│   └── GovSync/                           ← 新增目录
│       ├── IGovSyncService.cs             ← 新增
│       ├── XiaoShanGovSyncService.cs      ← 新增
│       └── GovSyncModels.cs               ← 新增
├── Models/
│   └── WeightSyncRecord.cs                ← 新增
└── EntityFrameworkCore/
    └── (现有 DbContext 扩展)              ← 修改：添加 DbSet

MaterialClient/
└── Services/
    └── WeightSyncBackgroundService.cs     ← 新增：后台定时同步
```

## 2. 分阶段实施

### 阶段一：LRP 图片保存（前置条件）

**原因**：同步功能必须附带图片，当前 `HikvisionLprService` 回调中不保存图片。

**改动文件**：`MaterialClient.Common/Services/Hikvision/HikvisionLprService.cs`

**改动内容**：

1. 在 `HandlePlateResult` 和 `HandleItsPlateResult` 中提取图片字节数据
2. 保存到本地文件系统（路径格式：`picture/{yyyy-MM-dd}/{timestamp}_{index}.jpg`）
3. 将图片路径列表附加到 `LicensePlateRecognizedEventData` 事件中
4. 可选：从 `CaptureDevice.CompressImage` 移植压缩逻辑，超过阈值时压缩

**数据流变化**：
```
之前：LRP 回调 → 发布车牌号事件
之后：LRP 回调 → 保存图片 → 发布车牌号+图片路径事件
```

**风险**：低。仅扩展事件数据，不改变现有行为。

3. `XiaoShanGovSyncService.cs` — 萧山1 同步实现

核心逻辑（从 `ServicePushWeightToGovApi.cs` 提取）：

```csharp
public class XiaoShanGovSyncService : IGovSyncService
{
    // 定时器扫描未同步记录（5秒间隔）
    // 转换为 XiaoShanWeightRequest 格式
    // 从文件系统读取图片 → Base64 编码
    // HTTP POST 到 {govXiaoShanUrl}/post
    // 更新 SyncStatus
    // FailCount >= 9 && 非网络原因 → 标记"不再上传"
}
```

**关键移植点**：

| GovClient 代码 | 移植策略 |
|---------------|---------|
| `agentTmer_Main_Elapsed` 轮询逻辑 | 使用 `PeriodicTimer` 或 `IHostedService` |
| `GetImages()` 图片读取 | 提取为独立方法，使用 `IHttpClientFactory` |
| 萧山1 字段映射 | 直接移植 `carType`/`deviceID`/`siteType`/`goodsWeight` 赋值逻辑 |
| `Post()` HTTP 请求 | 替换为 `HttpClient.PostAsJsonAsync` |
| `SaveToSqlLite()` 状态更新 | 替换为 EF Core `DbContext.SaveChanges` |
| `GovApiOnline` 网络检测 | 可简化为 try-catch HTTP 请求 |

**不需要移植**：
- 心跳包逻辑（萧山1 不需要）
- RSA 授权检查（MaterialClient 有自己的授权机制）
- 联通/电信模式分支

**风险**：中。HTTP 接口对接需要测试。

### 阶段四：后台定时服务

### 阶段五（扩展）：客户端与设备在线状态上报

**优先级**：低。独立于核心同步流程，可后续单独实施。

**需求**：服务端需要知道客户端是否在线，以及客户端连接的各设备（地磅、相机）是否在线。

#### 现状

- **GovClient**：通过 `ServiceTelnet`（1 秒 Telnet 探测）和 `Form1`（3 秒定时器）在本地检测网络和设备状态，但**不向服务端上报**。
- **MaterialClient**：已有 `LprDeviceOnlineStatusService` 统一查询设备在线状态，`MinimalWebHostService` 提供回调端点接收设备心跳，但同样**不向服务端汇总上报**。
- **XiaoShanServe**：当前没有接收在线状态的 API 端点。

#### 方案设计

**客户端侧**（MaterialClient 新增）：

新增一个后台服务，定期采集本地设备状态并向服务端上报。

```
MaterialClient.Common/
└── Services/
    └── GovSync/
        ├── ClientHeartbeatService.cs   ← 新增：定时采集并上报
        └── ClientStatusReport.cs       ← 新增：状态上报模型
```

```csharp
/// <summary>
/// 客户端状态上报模型
/// </summary>

{

```

#### 与核心同步的关系

```
独立模块，不干扰称重同步主链路：

称重同步（阶段一~四）：    地磅 → 稳定 → 入库 → 定时上报 → API/post
在线状态上报（阶段五）：   定时采集 → API/heartbeat → 服务端存储
                          ↑ 完全独立，失败不影响称重数据上报
```

#### 复用说明

| 现有能力 | 复用方式 |
|---------|---------|
| `LprDeviceOnlineStatusService.IsOnline()` | 直接调用，获取相机在线状态 |
| `GovSyncOptions` 配置 | 扩展 `HeartbeatIntervalSeconds` 和 `HeartbeatApiUrl` |
| `IHttpClientFactory` | 复用，用于发送心跳请求 |
| XiaoShanServe 现有框架 | 扩展 Controller + SqlSugar 实体 |

#### 配置项扩展

```json
{
  "GovSync": {
    "HeartbeatEnabled": false,
    "HeartbeatIntervalSeconds": 30,
    "HeartbeatApiUrl": "http://191.12.234.212:8899/api/heartbeat"
  }
}
```

默认关闭（`HeartbeatEnabled: false`），需要时在配置中启用。

**风险**：低。独立模块，失败不影响核心功能，可随时关闭。

## 3. 配置项

在 MaterialClient 的 `appsettings.json` 中新增：

```json
{
  "GovSync": {
    "Enabled": true,
    "ApiUrl": "http://191.12.234.212:8899/api/post",
    "BuildLicenseNo": "",
    "PollIntervalSeconds": 5,
    "MaxFailCount": 9,
    "ImageCompressThresholdKB": 200,
    "HeartbeatEnabled": false,
    "HeartbeatIntervalSeconds": 30,
    "HeartbeatApiUrl": "http://191.12.234.212:8899/api/heartbeat"
  }
}
```

对应配置类：

```csharp
public class GovSyncOptions
{
    public bool Enabled { get; set; }
    public string ApiUrl { get; set; }
    public string BuildLicenseNo { get; set; }
    public int PollIntervalSeconds { get; set; } = 5;
    public int MaxFailCount { get; set; } = 9;
    public int ImageCompressThresholdKB { get; set; } = 200;

    // 阶段五：在线状态上报
    public bool HeartbeatEnabled { get; set; } = false;
    public int HeartbeatIntervalSeconds { get; set; } = 30;
    public string HeartbeatApiUrl { get; set; }
}
```

## 4. 风险评估

| 风险 | 等级 | 说明 | 缓解 |
|------|------|------|------|
| LRP 回调中保存图片影响性能 | 低 | 图片 I/O 在 SDK 线程执行 | 异步保存，不阻塞回调 |
| 政府 API 地址/格式变更 | 中 | 萧山 API 无官方文档 | 抽象接口，配置化 URL |
| 图片路径在同步时不存在 | 中 | 图片可能被清理 | 同步前检查文件存在性，缺失时标记"数据缺失" |
| 与现有海康威视服务冲突 | 低 | 仅扩展回调，不改核心逻辑 | 图片保存为可选行为，通过配置开关控制 |

## 5. 与已有调研的关系

`docs/2026-05-12-govclient-weight-to-xiaoshanserve/` 中的调研结论：

- **方向 1（迁移到 MaterialClient）**是推荐方案，本方案是其具体化
- **不需要进出场匹配**（本次调研前提），降低了复杂度
- **不需要 XiaoShanServe 中转**：MaterialClient 可以直连萧山 API，减少一层故障点
- GovClient 的代码质量问题（R1-R9）在本方案中通过使用 MaterialClient 架构自动规避
