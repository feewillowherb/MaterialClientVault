# Urban 客户端 PFX 授权方案调研

## 调研背景

Urban 客户端当前使用简单的字符串验证机制（BuildLicenseNo 和 FdBuildLicenseNo）进行授权。为了增强安全性，调研是否可以引入 PFX（Personal Information Exchange）证书方案，同时保留现有的验证流程。

## 调研目标

1. 分析 Urban 客户端当前授权机制
2. 评估 PFX 方案的可行性
3. 设计 PFX 实现方案，保留 BuildLicenseNo、FdBuildLicenseNo 验证流程

## 调研范围

- **UrbanManagement 项目**：ABP 框架的城管管理系统
- **MaterialClient 项目**：现有的材料客户端，使用 Refit + Bearer Token 授权
- **Fdsoft.Weight.GovClient 项目**：原有的政府客户端，提供参考实现

## 关键发现

### 1. 当前授权机制分析

#### 1.1 UrbanManagement 验证机制

```csharp
// 实体定义
public class GovProject : Entity<Guid>
{
    public string ProName { get; set; } = default!;
    public string? BuildLicenseNo { get; set; }        // 建设许可证号
    public string? FdBuildLicenseNo { get; set; }      // 对接码
}
```

**验证逻辑**：
- 在 API 接收端验证 `BuildLicenseNo` 或 `FdBuildLicenseNo`
- 通过字符串匹配查询数据库中的项目信息
- 双重验证：先验证对接码，再验证建设许可证号

#### 1.2 MaterialClient 授权机制

```csharp
// 使用 Bearer Token 授权
public class MaterialPlatformBearerTokenHandler : DelegatingHandler
{
    protected override async Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request, CancellationToken cancellationToken)
    {
        var token = session?.AccessToken;
        if (!string.IsNullOrWhiteSpace(token))
            request.Headers.Authorization = new AuthenticationHeaderValue("Bearer", token);
        return await base.SendAsync(request, cancellationToken);
    }
}
```

**授权特点**：
- 基于 Refit 框架的 HTTP 客户端
- Bearer Token 认证
- 支持用户名密码登录
- 机器码验证

### 2. 安全性分析

#### 2.1 当前机制的安全风险

| 风险类型 | 描述 | 严重程度 |
|---------|------|----------|
| 字符串泄露 | 许可证号直接存储在数据库和配置文件中 | 高 |
| 传输安全 | 未使用加密传输 | 中 |
| 重放攻击 | 缺乏请求签名机制 | 中 |
| 身份伪造 | 容易伪造许可证号 | 高 |

#### 2.2 PFX 方案的安全优势

| 优势 | 描述 |
|------|------|
| 私钥保护 | 私钥安全存储在证书文件中 |
| 双向认证 | 服务器和客户端互相验证 |
| 传输加密 | SSL/TLS 加密通信 |
| 请求签名 | 数字签名防止篡改 |
| 证书吊销 | 支持证书撤销机制 |

### 3. 技术可行性评估

#### 3.1 .NET 环境支持

```csharp
// .NET 提供完善的证书支持
using System.Security.Cryptography.X509Certificates;

// 1. 加载 PFX 证书
var certificate = new X509Certificate2("client.pfx", "password");

// 2. 配置 HttpClient 使用证书
var handler = new HttpClientHandler();
handler.ClientCertificates.Add(certificate);

// 3. 与 Refit 集成
services.AddRefitClient<IUrbanManagementApi>()
    .ConfigureHttpClient(c =>
    {
        c.BaseAddress = new Uri(apiUrl);
    })
    .ConfigurePrimaryHttpMessageHandler(() => new HttpClientHandler
    {
        ClientCertificates = { certificate }
    });
```

#### 3.2 ABP 框架集成

ABP 框架支持灵活的 HTTP 客户端配置，可以无缝集成 PFX 方案：
- 支持 `HttpClientFactory` 扩展
- 支持自定义 `DelegatingHandler`
- 支持配置文件管理

### 4. 兼容性保证

#### 4.1 保留现有验证流程

```csharp
// 双重验证策略
public class PfxLicenseValidator
{
    public async Task<ValidationResult> ValidateAsync(
        X509Certificate2 certificate,
        string? buildLicenseNo,
        string? fdBuildLicenseNo)
    {
        // 1. 证书验证（新增）
        var certResult = await ValidateCertificateAsync(certificate);
        if (!certResult.IsValid)
            return certResult;

        // 2. 许可证号验证（保留）
        var licenseResult = await ValidateLicenseNoAsync(buildLicenseNo, fdBuildLicenseNo);
        if (!licenseResult.IsValid)
            return licenseResult;

        // 3. 关联验证（证书与许可证号关联）
        return await ValidateAssociationAsync(certificate, buildLicenseNo);
    }
}
```

## 结论

### 可行性评估

✅ **PFX 方案完全可行**

1. **技术成熟度**：.NET 提供完善的证书支持
2. **框架兼容性**：ABP + Refit 完美支持
3. **向后兼容**：可保留现有 BuildLicenseNo、FdBuildLicenseNo 验证
4. **渐进迁移**：支持逐步迁移，降低风险

### 推荐方案

采用 **PFX 证书 + 许可证号双重验证** 的混合方案：
1. 主要验证：PFX 证书（高安全性）
2. 辅助验证：BuildLicenseNo、FdBuildLicenseNo（兼容性）
3. 关联验证：证书与许可证号的绑定关系
