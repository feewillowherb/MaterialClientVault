## 🔍 调研发现

### 当前授权机制分析

#### UrbanManagement 验证机制
- **验证方式**：BuildLicenseNo 和 FdBuildLicenseNo 字符串验证
- **验证流程**：
  1. 接收 API 请求中的许可证号
  2. 在数据库中查找匹配的项目
  3. 验证通过后处理业务逻辑
- **安全问题**：字符串易泄露、无传输加密、缺乏身份验证

#### MaterialClient 授权参考
- **授权方式**：Bearer Token + Refit HTTP 客户端
- **架构特点**：
  - 使用 Refit 框架生成类型安全的 HTTP 客户端
  - Bearer Token 通过 `DelegatingHandler` 自动添加
  - 支持用户名密码登录和机器码验证
- **可复用组件**：`MaterialPlatformBearerTokenHandler`、Refit 配置模式

### PFX 方案技术优势

| 特性 | PFX 方案 | 现有方案 |
|------|----------|----------|
| **双向认证** | ✅ 支持 | ❌ 不支持 |
| **传输加密** | ✅ SSL/TLS | ❌ 明文传输 |
| **数字签名** | ✅ 支持 | ❌ 不支持 |
| **证书吊销** | ✅ 支持 | ❌ 不支持 |
| **私钥保护** | ✅ 高安全性 | ❌ 无保护 |
| **标准化** | ✅ 行业标准 | ❌ 自定义实现 |

## 🛠️ 实施方案

### 架构设计
```
客户端: PFX证书 + 许可证号 → HTTP客户端 → Refit API
服务端: 证书验证中间件 + 双重验证服务 → 业务逻辑
```

### 核心组件

#### 客户端组件
1. **PfxCertificateManager**：证书加载和验证
2. **PfxAuthorizationHandler**：授权处理器
3. **Refit API 配置**：支持证书的 HTTP 客户端

#### 服务端组件
1. **ClientCertificateMiddleware**：证书验证中间件
2. **DualValidationService**：双重验证服务
3. **API 兼容层**：保留现有 API 接口
