## 关键技术发现

1. **XiaoShanServe 是轻量中转层**：仅含 6 个 Controller、3 个核心实体、1 个后台服务，功能边界清晰
2. **MaterialClient 已验证 ABP 模式**：使用 Volo.Abp.Core + EF Core SQLite + Autofac + AsyncPeriodicBackgroundWorkerBase
3. **API 兼容可行**：XiaoShanServe 的 `POST /Api/Post` 接口输入输出格式简单，在 ABP 中通过兼容控制器即可保持
4. **后台同步可替换**：`ExplortStatisticBgService`（5秒轮询）可映射为 ABP `AsyncPeriodicBackgroundWorkerBase`
5. **图片存储需重构**：从逗号分隔本地路径改为 `AttachmentFile` 体系（MaterialClient 已有成熟实现）
6. **新增实体已就位**：`UrbanWeighingRecord` 是新客户端（MaterialClient 架构）的接收端实体，已含幂等去重（ClientRecordId）
7. **JSON 序列化已统一**：`UrbanManagementAppModule` 中已配置 `JsonNamingPolicy.CamelCase`，与旧系统一致

## 风险概览

| 风险 | 等级 | 说明 |
|------|------|------|
| 旧客户端 API 未兼容 | **高** | 旧 GovClient 的 `POST /Api/Post` 未实现，无法对接旧客户端 |
| ORM 差异 | 中 | SqlSugar → EF Core 语法差异，需逐条对照查询 |
| 后台服务迁移 | 中 | BackgroundService → ABP Background Worker，生命周期管理不同 |
| Mock 数据未替换 | 中 | ProjectController/SyncInfoController 返回 Mock 数据，需接入真实数据库 |
| 实体字段差异 | 低 | GovProject 主键实际为 Guid（非规划的 int），GovSyncData/GovLog 主键为 int（非规划的 long） |
| 配置管理 | 低 | 当前 AppSettings 极简，需扩展为 IOptions<T> 模式 |
