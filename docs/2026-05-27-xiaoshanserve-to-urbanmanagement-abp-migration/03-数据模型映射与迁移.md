# 03 - 数据模型映射与迁移

> **更新说明**：本文档已根据 UrbanManagement 实际实体定义更新。实体设计策略为**保留旧字段名**（而非原规划的重命名方案），新增 `UrbanWeighingRecord` 实体。

## 1. 实体映射总览

### 1.1 实际实现策略

与原规划不同，实际实现采用**保持旧字段名**的策略：
- 旧系统字段名在 EF Core 实体中完全保留
- 无 `[Column("旧字段名")]` 映射需求
- 减少迁移风险，旧数据库可直接附加

### 1.2 Gov_Project → GovProject

| 旧字段（SqlSugar） | EF Core 属性 | 类型 | 变更 | 说明 |
|-------------------|-------------|------|------|------|
| *(无)* | **Id** (Guid) | 新增 | PK 改为 Guid | ProId 即为 Id，不再单独字段 |
| ProId | *(合并到 Id)* | 删除 | ProId = Id | 简化设计 |
| ProName | ProName | string | 不变 | |
| BuildLicenseNo | BuildLicenseNo | string? | 不变 | |
| FdBuildLicenseNo | FdBuildLicenseNo | string? | 不变 | |
| SyncStatus | SyncStatus | bool? | 不变 | 可空类型 |
| LastSyncTime | LastSyncTime | DateTime? | 不变 | |

**与原规划的关键差异**：
1. **主键类型**：原规划 `int`，实际 `Guid` — ProId 即主键，简化数据模型
2. **字段名保留**：原规划建议重命名（`DeleteStatus→IsDeleted`、`AddTime→CreationTime`），实际全部保留旧名
3. **无审计接口**：原规划建议 `IHasCreationTime`，实际未采用

### 1.6 新增：Urban_WeighingRecord（全新实体）

| 属性 | 类型 | 说明 |
|------|------|------|
| Id | long | 自增主键 |
| ClientRecordId | long | 客户端记录 ID（**唯一索引**，幂等去重） |
| PlateNumber | string? | 车牌号 |
| TotalWeight | decimal | 总重量（decimal 精度） |
| WeighingTime | DateTime | 称重时间 |
| AddTime | DateTime | 入库时间 |
| SyncType | int? | 同步类型 |

**设计说明**：
- 这是**全新实体**，不在旧系统中存在
- 用于接收新客户端（MaterialClient 架构）上传的称重记录
- `ClientRecordId` 唯一索引确保幂等性
- `TotalWeight` 使用 `decimal`（旧系统用 `int`/`string`），精度更高
- 字段命名使用英文标准命名（`PlateNumber` 而非 `CarNo`），与 MaterialClient 风格一致

## 2. 实际 EF Core 配置

```csharp
// UrbanManagementDbContext.cs OnModelCreating（实际代码）

// Gov_Project 表
builder.Entity<GovProject>(b =>
{
    b.ToTable("Gov_Project");
    b.HasKey(e => e.Id);                         // Guid 主键
    b.Property(e => e.ProName).IsRequired().HasMaxLength(200);
    b.Property(e => e.BuildLicenseNo).HasMaxLength(200);
    b.Property(e => e.FdBuildLicenseNo).HasMaxLength(200);
    // 无额外索引配置（原规划建议对 BuildLicenseNo/FdBuildLicenseNo 建索引）
});

// Gov_SyncData 表
builder.Entity<GovSyncData>(b =>
{
    b.ToTable("Gov_SyncData");
    b.HasKey(e => e.Id);                         // int 主键
    b.Property(e => e.CarNo).HasMaxLength(50);
    b.Property(e => e.GoodsWeight).HasMaxLength(50);
    b.Property(e => e.SnapTime).HasMaxLength(100);
    b.Property(e => e.SnapImages).HasMaxLength(2000);
    // 无额外索引配置
});

// Gov_Log 表
builder.Entity<GovLog>(b =>
{
    b.ToTable("Gov_Log");
    b.HasKey(e => e.Id);                         // int 主键
    b.Property(e => e.SyncMsg).HasMaxLength(2000);
    b.Property(e => e.SyncResult).HasMaxLength(2000);
});

// Urban_WeighingRecord 表（新增）
builder.Entity<UrbanWeighingRecord>(b =>
{
    b.ToTable("Urban_WeighingRecord");
    b.HasKey(e => e.Id);                         // long 主键
    b.Property(e => e.PlateNumber).HasMaxLength(50);
    b.Property(e => e.SnapImages).HasMaxLength(2000);
    b.HasIndex(e => e.ClientRecordId).IsUnique(); // 幂等去重索引
});
```

## 4. 数据库迁移策略（更新）

### 4.1 当前状态

- 使用 SQLite，连接字符串：`Data Source=UrbanManagement.db`
- `AddDefaultRepositories(true)` 自动注册默认仓储
- 支持 EF Core Migrations（尚未生成初始迁移）

### 4.2 迁移策略（保持表名兼容）

由于实际实体**保留了旧字段名**，迁移策略比原规划更简单：

| 场景 | 处理方式 |
|------|---------|
| 旧数据库直接附加 | GovProject/GovSyncData/GovLog 字段完全兼容 |
| GovProject.Id 列 | 旧表 ProId 列需要映射（旧表无 Id 列，ProId 即业务主键） |
| 新增 Urban_WeighingRecord 表 | 自动创建，不影响旧表 |
| 新增字段（如有） | 通过 EF Core Migration 添加 |

### 4.3 GovProject 主键迁移

GovProject 的主键从旧系统的 `ProId (string/GUID)` 改为 `Id (Guid)`，这是最大的迁移挑战：

**方案 A**：`[Column("ProId")]` 映射
```csharp
// 将 Id 映射到旧表的 ProId 列
b.Property(e => e.Id).HasColumnName("ProId");
```

**方案 B**：保留 ProId 作为独立字段
```csharp
// 增加自增 int 主键，ProId 保留为业务字段（回到原规划）
public class GovProject : Entity<int>
{
    public string ProId { get; set; }  // GUID 业务 ID
    // ...
}
```

> **建议**：根据旧数据库实际结构决定。如果旧表已有 ProId 列且值均为 GUID，方案 A 更简洁。
