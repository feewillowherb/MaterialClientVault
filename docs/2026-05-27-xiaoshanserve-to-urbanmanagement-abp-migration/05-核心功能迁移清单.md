## 3. 逐项迁移要点（更新）

### F1: 数据接收端点（❌ 未实现）

**当前状态**：未创建 `LegacyApiController`，旧客户端 `POST /Api/Post` 无法使用。

**已完成的基础设施**：
- ✅ JSON 序列化已配置 camelCase
- ✅ `GovSyncData` 实体已定义
- ✅ `GovProject` 实体已定义（含 BuildLicenseNo/FdBuildLicenseNo）
- ❌ 接入码验证逻辑未实现
- ❌ 图片处理服务未实现

**迁移要点**：
- `dynamic` 参数类型保持，确保旧客户端 JSON 不被拒绝
- grossWeight > 0 覆盖 goodsWeight 的逻辑必须保留
- 响应格式必须包含 `code` 字段（当前新 API 缺少此字段）

### F2: 接入码双重验证（❌ 未实现）

**当前状态**：无任何验证逻辑。

**已具备**：
- ✅ `GovProject.BuildLicenseNo` 字段
- ✅ `GovProject.FdBuildLicenseNo` 字段

**待实现**：
- `IGovProjectManager` 领域服务或在 AppService 中实现
- fdBuildLicenseNo（凡东码）和 buildLicenseNo（城管码）二选一验证逻辑
- 验证失败时的错误消息格式保持与旧系统一致

### F3+F4: 图片处理（❌ 未实现）

**当前状态**：无图片处理逻辑。

`UrbanWeighingRecord` 不再使用 `SnapImages` 字段。旧客户端发送 Base64 需转换为 `AttachmentFile` 关联数据（仅 `Lrp`/`UrbanPhoto`）。

**迁移要点**：
- 旧逻辑：Base64 → byte[] → 压缩（>200KB）→ 保存到 `FilesPhysicalPath/TempUpload/{buildLicenseNo}/{ticks}_{i}.jpg`
- 新逻辑：Base64 → `AttachmentFile`（`Id: Guid`, `AttachType` 仅 `Lrp`/`UrbanPhoto`）+ `UrbanWeighingRecordAttachment` 关联
- 旧 `SnapImages` 历史数据迁移时全部按 `AttachType=Lrp` 导入

### F6: 后台同步服务（❌ 未实现）

**当前状态**：无后台服务。

**迁移要点**：
- `BackgroundService` → `AsyncPeriodicBackgroundWorkerBase`
- `HttpWebRequest` → Refit `IGovSyncHttpClient`
- 重试逻辑由 Polly 策略处理
- 5 秒轮询间隔保留
- RetryCount >= 10 → 不再同步（保留，状态维护在 UrbanWeighingRecord）

### F9-F14: 管理功能（⚠️ Mock 实现）

**当前状态**：`ProjectController` 和 `SyncInfoController` 的所有方法都通过 `SampleDataProvider` 返回硬编码 Mock 数据。

**待实现**：
- 接入真实 `IRepository<GovProject, Guid>` 进行 CRUD
- 接入真实 `IRepository<GovSyncData, Guid>` 进行查询
- 接入真实 `IRepository<GovLog, Guid>` 进行日志查询

### F20-F21: 称重记录 API（✅ 已实现 — 新功能）

这是原规划中没有的新功能，用于新客户端（MaterialClient 架构）上传称重数据。

**已实现**：
- `POST /api/urban/weighing-records` — 接收称重记录
  - `ClientRecordId` 幂等去重
  - `ModelState` 验证
  - 错误处理
- `GET /api/urban/weighing-records` — 分页查询
  - 车牌号模糊搜索
  - 时间范围筛选
  - 分页排序
